【Golang】DEX 扫块总结

扫块

扫块即解析每个区块内的每笔交易，识别出我们所需要的交易，并且告知对应的服务，处理对应的订单

我们的 扫块是脱胎于交易所扫块服务，所以很多设计之处会考虑到不同的模型：UTXO 账户模型 tag模型，这里只关注账户模型

一、第一个rpc 获取Tip–>确认需要解析哪个块

二、第二个rpc（eth_getBlockByNumber、eth_getBlockReceipts、debug_traceBlockByNumber）—解析对应块

确认好需要同步的块高，开始处理区块

总共其实是三个部分 由3个goroutine 去处理

2.1 Block区块信息 eth_getBlockByNumber // 通过区块高度获取区块信息

  • 标准以太坊节点就有
  • 这个简单，直接请求RPC 的 eth_getBlockByNumber 然后用 Block结构体接收即可
  • 数据量很小

这个很基础，就是获取整个区块的信息

2.2 Receipts信息 瑞思普斯 eth_getBlockReceipts 交易执行后的结果凭证，包含交易的执行状态和结果。

  • 请求RPC 的eth_getBlockReceipts 方法

  • 标准以太坊节点就有

  • 交易收据：包含 gas、日志、状态码

  • 数据量中等

-

2.3 Traces原始信息 ：获取区块内所有交易的详细执行过程（追踪）

  • 请求RPC 的debug_traceBlockByNumber
  • 需要 debug API节点
  • 完整调用栈、内存、存储
  • 数据量很大

将返回的数据递归处理后：

汇总

最后所有的数据汇总成一个结构体

[!IMPORTANT]

这三个方法都是通过块高 即可解析出此块内的所有交易的Receipts 和trace 而不需要遍历块内的所有交易哈希，可以减少很多rpc的调用，但是当个rpc方法的数据量则特别大

三、处理获取到的块内信息—->行为分类

走到这一步，单个块内的所有数据已经拿到，现在需要将数据进行处理，

  • 1、解析区块内所有交易，提取行为：

    • 转账、

    • Swap、

    • Mint/Burn、

    • 池子流动性、

    • 新Token

  • 2、并发获取链上数据

    • balance.Parse() 获取账户余额快照
    • tokenchecker.Parse() 获取Token信息

  • 3、数据增强（Enhancement）

    • 转账数据增强
    • Swap数据增强
    • Mint/Burn数据增强
    • 流动性数据增强
    • 新Token数据增强
    • 计算Swap的余额变化

  • 4、数据清理和去重

    • 移除Swap交易中的转账重复数据

最后转化成业务可用数据

3.1 区块内行为提取之转账

怎么从我们解析出来的Receipt和Trace 里面区分出转账来呢

  1. traceParse - 解析 Internal Traces (内部调用追踪)

    1. 获取 Native Token 转账 (ETH/BNB 等主网币)
    2. 包括合约内部的原生币转账

  2. logParse - 解析 Event Logs (事件日志)

  3. 获取 ERC20/ERC404 Token 转账

  4. 通过 Transfer 事件识别

  5. txParse - 解析交易本身

    1. 获取交易级别的余额变化
    2. 主要是 Gas 费用和直接转账的 value

3.2.1 fourmeme 内盘

fourmeme的机制是什么？

Fourmeme 是一个内盘交易平台 (Bonding Curve)：

用户想买一个新的Toekn (比如 MEME)

  • 用户转账给fourmeme合约，
  • fourmeme抛出 emit TokenPurchase(MEME, user, 100 MEME, 1000 USDC)事件
  • fourmeme 转meme给用户
  • 卖单也是一样，卖单抛出emit TokenSale(MEME, user, 100 MEME, 900 USDC, 100fee) 事件

fourmeme的其他特点

  • 没有独立的流动性池 (Pool)
  • 价格由 Bonding Curve 公式决定
  • 所有交易都通过 FourmemeContract 进行
  • 用 Token 合约地址作为 “Pool” 标识
